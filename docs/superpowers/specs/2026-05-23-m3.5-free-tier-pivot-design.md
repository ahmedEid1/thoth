# Atlas — M3.5 Free-Tier Pivot Design Spec

**Status:** Approved for implementation planning
**Date:** 2026-05-23
**Author:** Ahmed Hobeishy (with Claude)
**Supersedes:** M3.5 (Hetzner deploy) section of `2026-05-22-atlas-design.md` §15 Roadmap
**Builds on:** `v0.3.0-m3` (4-node LangGraph + HITL gates, Anthropic-only, dev-local stack)

---

## 1. Purpose

Atlas v0.3.0-m3 ships a working agent end-to-end but is bound to two paid assumptions: **Anthropic-only LLM** and **Hetzner self-host deploy**. This spec retargets both to a **$0/month** stack — free LLM by default with a one-flag swap to paid providers, and a fully free PaaS deploy — while keeping a self-host escape hatch documented for the day we outgrow free tiers.

Constraints driving the redesign:
- Budget: **$0/month** for at least the next 60 days
- LLM: free default with `LLM_PROVIDER` config to swap to paid (Anthropic, OpenAI, Groq)
- Deploy: live URL reachable from the public internet without spend
- Portfolio value preserved: still demonstrates LangGraph, HITL, evals, observability — the M1/M2/M3 hiring signals must all remain visible

---

## 2. Why this pivot now

- M3 shipped 6 weeks ahead of the original 2026-07-10 timeline. We have runway to refactor without slipping M6.
- The cleanest CV signal is "shipped a live demo." A free PaaS deploy beats a Hetzner box that needs a credit card to provision.
- Vercel AI SDK is the 2026 standard for provider-agnostic LLM calls in TypeScript apps. Adopting it now also tracks current industry practice for the JD's `tools/frameworks` requirement.
- Google Gemini 2.5 Flash's free tier is generous enough (15 RPM, 1500 RPD, 1M-token context) to run real end-to-end reviews without ever hitting a paywall.

## 4. Architecture

### 4.1 LLM abstraction (the chokepoint)

`lib/llm.ts` keeps its public surface (`runLLM<T>(args): Promise<RunLLMResult<T>>`) so all 5 prompt callers compile unchanged. Internally it becomes a dispatcher.

```
lib/llm.ts                       ← public API. Dispatches to provider.
lib/llm/
  providers/
    index.ts                     ← registry; reads LLM_PROVIDER env
    gemini.ts                    ← @ai-sdk/google adapter (DEFAULT)
    anthropic.ts                 ← @ai-sdk/anthropic adapter
    openai.ts                    ← @ai-sdk/openai adapter
    groq.ts                      ← @ai-sdk/groq adapter
  tiers.ts                       ← maps "smart"|"fast" → model id per provider
  telemetry.ts                   ← Langfuse OTel wiring (replaces direct lf.trace calls)
```

**Tier mapping** (`lib/llm/tiers.ts`):

| Tier  | Gemini (default)        | Anthropic           | OpenAI       | Groq                    |
|-------|------------------------ |---------------------|--------------|-------------------------|
| smart | `gemini-2.5-pro`        | `claude-opus-4-7`   | `gpt-4o`     | `llama-3.3-70b-versatile` |
| fast  | `gemini-2.5-flash`      | `claude-sonnet-4-6` | `gpt-4o-mini`| `llama-3.1-8b-instant`  |

**Each prompt declares a tier, not a model.** E.g., planner = `smart`, retriever per-paper score = `fast`, assessor claim extraction = `smart`, drafter = `smart`, summarize-paper = `fast`. Swapping provider keeps relative quality intent intact.

`RunLLMArgs<T>` gains one new required field: `tier: "smart" | "fast"`. All 5 existing prompt callers update accordingly (one-line change each).

### 4.2 Call shape (Vercel AI SDK)

`runLLM` calls `generateObject` under the hood, with telemetry attributes that Langfuse picks up:

```ts
import { generateObject } from 'ai';
import { resolveProvider } from '@/lib/llm/providers';
import { resolveTier } from '@/lib/llm/tiers';

const provider = resolveProvider(env.LLM_PROVIDER); // factory function
const model = provider(resolveTier(args.tier, env.LLM_PROVIDER));

const { object, usage } = await generateObject({
  model,
  schema: args.schema,                 // Zod schema, unchanged
  system: args.system,
  messages: args.messages,
  experimental_telemetry: {
    isEnabled: true,
    functionId: args.functionId,       // e.g., "plan-review"
    metadata: {
      runId: args.runId,
      projectId: args.projectId,
      userId: args.userId,
      sessionId: args.runId,
      tags: [args.tier, env.LLM_PROVIDER],
    },
  },
});
```

The `experimental_telemetry.metadata` gets picked up automatically by `LangfuseSpanProcessor` and mapped to Langfuse trace fields (`langfuseTraceId`, `sessionId`, `userId`, `tags`). Cost & token usage land on the trace via the same exporter.

### 4.3 Telemetry wiring (Next.js 16 + Vercel)

New file: `instrumentation.ts` at the Next.js project root.

```ts
import { registerOTel } from '@vercel/otel';
import { LangfuseExporter } from 'langfuse-vercel';

export function register() {
  registerOTel({
    serviceName: 'atlas',
    traceExporter: new LangfuseExporter({
      environment: process.env.VERCEL_ENV ?? 'development',
    }),
  });
}
```

The `LangfuseExporter` reads `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, `LANGFUSE_HOST` from env. Same env names as M2 — flips between self-hosted (`localhost:3030`) and Langfuse Cloud (`https://cloud.langfuse.com`) by changing the host value only.

For long-running Trigger.dev tasks (where Next.js' serverless flush hook doesn't apply), we keep an explicit `await lf.flushAsync()` in a `finally` block. M2's pattern stays.

### 4.5 Prisma + Neon specifics

Per Neon's official Prisma guide, use **`@prisma/adapter-neon`** (NOT `@prisma/adapter-pg`) for serverless performance on Vercel. Replace the M1 `PrismaPg` adapter:

```ts
// lib/db.ts
import { PrismaClient } from '@/app/generated/prisma/client';
import { PrismaNeon } from '@prisma/adapter-neon';

const adapter = new PrismaNeon({
  connectionString: process.env.DATABASE_URL!, // pooled (-pooler suffix)
});

export const prisma = new PrismaClient({ adapter });
```

Env vars:
- `DATABASE_URL` → pooled connection (with `-pooler` in hostname) — used at runtime
- `DIRECT_DATABASE_URL` → direct connection — used by `prisma migrate` only (already wired in `prisma.config.ts`)

### 4.6 Object store (R2)

R2 is S3-compatible. The existing `lib/object-store.ts` (M1) already uses AWS SDK v3 with `forcePathStyle: true`. Only the env values change:

```env
S3_ENDPOINT="https://<account-id>.r2.cloudflarestorage.com"
S3_REGION="auto"
S3_ACCESS_KEY_ID="<r2-access-key>"
S3_SECRET_ACCESS_KEY="<r2-secret>"
S3_BUCKET="atlas-corpus"
S3_FORCE_PATH_STYLE="false"   ← virtual-hosted-style for R2
```

Zero code change. Just env values. Integration tests (M1 Task 5) keep pointing at local MinIO in dev/CI.

## 5. Self-host fallback (M3.5c)

Goal: zero-effort migration off Vercel if/when free tier limits start to bite. Atlas should remain a single `git pull && docker-compose up -d` deploy.

`infra/self-host/`:
- `docker-compose.prod.yml` — extends the dev `docker-compose.yml`. Adds:
  - Caddy reverse proxy with `caddy.atlas.example` auto-TLS
  - Postgres backup cronjob (daily pg_dump to a mounted volume)
  - `atlas` web service running `pnpm start`
  - Self-hosted Trigger.dev v4 (referencing their `docker-compose-trigger.dev` cookbook)
  - Keeps self-hosted Langfuse from M2 (already proven)
- `.env.prod.example` — production env template
- `Caddyfile` — single-domain config

`docs/self-host/oracle-cloud-quickstart.md`:
- Step-by-step on Oracle Cloud Always Free (Ampere A1: 4 ARM cores, 24 GB RAM, free forever)
- DNS pointing
- TLS via Caddy + Let's Encrypt
- Clerk webhook URL update
- Trigger.dev v4 self-host pointer
- Backup/restore commands

This is **documented, not deployed**. Acceptance criterion: a fresh Ubuntu 22.04 ARM box can host-Atlas in ≤ 60 min following the doc.

## 7. Test strategy

**Continued rule:** no real LLM calls in unit/integration tests. All `generateObject` mocked.

New tests:
- `lib/llm/providers/index.test.ts` — `LLM_PROVIDER=gemini` returns Gemini factory; `=anthropic` returns Anthropic; unknown throws.
- `lib/llm/tiers.test.ts` — tier mapping per provider; `smart` always returns a model id; throws for unknown tier.
- `lib/llm.test.ts` (rewrite) — `runLLM` dispatches via provider, passes Zod schema to `generateObject`, captures usage, includes telemetry metadata. Mocks `ai`.
- `lib/db.test.ts` — `PrismaNeon` adapter chosen when `DATABASE_URL` set.

Existing tests updated to swap mock from `@anthropic-ai/sdk` to `ai` (same `vi.hoisted()` pattern).

**Smoke test** (M3.5b, manual, real LLM):
- Boot deployed Atlas at `atlas-xxx.vercel.app`
- Create project + upload sample PDF
- Start review; approve plan gate; approve papers gate
- Verify draft renders with valid `[paper_id]` citations

- Document in `docs/m3.5b-smoke-test.md` with screenshots

**Quota tracking** (lightweight, not a feature):

## 9. Risks & mitigations

| Risk                                                  | Likelihood | Mitigation                                          |
|-------------------------------------------------------|------------|-----------------------------------------------------|
| Gemini's structured-output schema enforcement differs from Anthropic's, causing prompt failures | Medium | Use `generateObject` (validated by AI SDK with retries); add provider-conformance test per prompt; for any prompt that fails on Gemini, add explicit JSON-mode override |
| Vercel cold start on Node runtime > 2-3s             | Low         | Acceptable for a research tool. Agent runs go through Trigger.dev, so end-user only waits on lightweight enqueue routes |
| Vercel function timeout (300s on Hobby)              | Low         | All long-running agent work is in Trigger.dev tasks already (M3) |
| Gemini free quota exhausted by a busy demo day        | Low (1500 RPD = ~50 reviews/day) | `LLM_PROVIDER` flag flips to Anthropic in seconds; document this in README |
| Neon 0.5 GB storage limit                            | Low         | ~50+ projects fit. Pgvector embeddings deferred to M4+; if needed, defer to a dedicated Pinecone free tier or pause embeddings on free |
| Langfuse Cloud 50K obs/mo                             | Low         | Each review ≈ 30-50 spans. Supports ~1000 reviews/mo. Self-host fallback documented in M3.5c |
| Trigger.dev Cloud free tier rules change             | Medium      | M3.5c documents self-host migration; project code is unchanged when switching deploy target |
| Provider abstraction leaks Gemini-specific behavior into prompts (e.g., "thinking" param) | Medium | Tier abstraction keeps it minimal; provider adapter handles param translation/dropping; integration test asserts both providers receive valid args |
